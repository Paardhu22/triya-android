# Triya Manager — Android — Project Status

_Last updated: 16 July 2026_

The native Android companion to **Triya Manager** (the Next.js web app in
`../triya-manager`). The mobile app mirrors the web app's business logic,
terminology, visual language (warm ink/tan "Swiss" palette, Sora type, the
dashboard's blueprint styling) and screen flows, adapted for touch. All data
is served by a realistic in-memory mock layer shaped exactly like the web
app's queries/actions, so swapping to the real API touches one layer only.

---

## Completed features

**Authentication**
- Login with the web's account-picker + password flow, on the web login
  page's ink/editorial composition (overline, ghost "PG" glyph, sand accents)
- Role-aware property selection: single-property managers auto-select;
  admins pick from all properties (web `/select-property` equivalent)
- Session + property guards on every route group, sign-out everywhere

**Dashboard** (web blueprint style)
- Grid-paper stat cards with FIG annotations, mono `>` kickers, pseudo-code
  footers, animated count-up figures and occupancy bars
- Capacity / occupancy / collections (month) / expenses (month) — flat
  properties count flats, PGs count beds (web rule)
- Room-capacity (sharing) or flat/block breakdown with per-row occupancy bars
- Recent payments and complaints panels with uppercase enum chips

**Floor Manager**
- Block selector (only when the property has >1 block — web rule), floor
  chips, room grid with per-bed visual status dots (Paid/Pending/Overdue/
  Vacant — the web's `bed-status.ts`, exact colors)
- Flat properties: tile = the unit, left-edge status accent, opens the unit
  sheet directly; ghost floor numeral (the web floor banner's giant number)
- Room detail with bed cards → BedSheet: tenant summary, mark paid, send
  invoice, record notice (15-day rule preview), vacate (deposit outcome
  computed via the shared `resolveDepositStatusOnVacate` rule)
- Check-in form for vacant beds: rent, maintenance, deposit (net of the
  ₹1,000 maintenance reserve), first-cycle status + payment method
  (cash/online/split with validated amounts)

**Tenants**
- List: search name/phone, All/Current/Past filter with counts, name/recent
  sort, result-count footer
- Profile: identity header with Current/Past marker, mark cycle paid/unpaid,
  send house rules (WhatsApp, gated on rules being set — web rule), delete
  with confirm, KYC & personal details, stay history with deposit/rent-cycle
  due text/leaving date, payment history with method + split breakdown +
  recorded-by, KYC document section

**Collections & Invoices**
- Collected/Outstanding summary tiles; Dues / Invoice History segments
- Dues: search, All/Pending/Overdue/Paid filter, per-row remind / mark paid /
  send invoice; bulk "Remind all" with confirmation
- Mark-paid flow: confirm amount + method (split validated to the paisa),
  then the web's "send invoice now?" follow-up chain
- Send-invoice sheet: previous due / extra charges (+label) / discount /
  notes over the shared totals math
  (`total = rent + maintenance + previous + extra − discount`, floored at 0),
  live preview, INV-YYYYMM-NNNN numbering, sent/failed status + resend

**Complaints**
- List: search, status + priority filters, counts; new-complaint sheet
  (title, description, priority, assignee) — complaints start OPEN
- Detail: inline status/priority/assignee management (resolving stamps
  `resolvedAt`), reporter and flat-aware location

**Expenses**
- List: search vendor/notes, category filter, filtered total; add-expense
  sheet with the web's category/subcategory requirement rule, amount, date,
  vendor, notes
- Detail: full facts + recorded-by, delete with confirmation

**Notifications** — unread styling, mark-one/mark-all read, unread count,
generated by mock actions (payments, check-ins, complaints…)

**Settings & Profile** — account identity card, working change-password
sheet (min 8 chars, verifies current), property details with ADMIN-only
editing (name/address/city/phone/house-rules), about + sign out

**Platform polish**
- Sora brand font (all weights) behind the native splash; warm web palette
  as the single token source; light-only (web parity)
- Motion system: screen/header entrances, capped list stagger, animated
  progress bars and counters, natural press scaling (Button/FAB), eased
  bottom sheets, shift-animating tabs, slide-from-right stack
- Skeleton loading on every data screen, wired error states with retry,
  distinct "empty vs. no-match" copy everywhere, pull-to-refresh on all lists

## Architecture

```
UI (screens + components)
  → hooks (useAsyncData/useQueries/useAction — React-Query-shaped)
    → src/mocks (queries.ts + actions.ts over an in-memory DB, ActionResult
      pattern, subscribe-on-mutate invalidation)
      → [swap point] real API client
```

- **Routing**: Expo Router file routes; `(auth)` group (ink screens) and
  `(main)` group (guarded shell: 5 tabs + pushed detail screens with native
  headers)
- **State**: React contexts — `AuthContext` (session), `PropertyContext`
  (active property, auto-select rule), `ThemeContext` (dark-mode seam)
- **Money**: integer paise everywhere (web convention), `formatINR`
- **Business rules** shared in `src/utils/tenancy.ts` (15-day notice, ₹1,000
  maintenance reserve, deposit resolution, cycle due text) — ported verbatim
  from the web's `lib/tenancy.ts`
- **Statuses**: `src/constants/status.ts` (app-wide) + `bedStatus.ts` (floor
  manager visual collapse) — same split as the web

## API integration points

Every screen consumes data through `src/hooks/useQueries.ts` (reads) and
`src/mocks/actions.ts` (writes returning `ActionResult`). To integrate:

1. Implement an API client with the same function signatures as
   `src/mocks/queries.ts` / `actions.ts` (they mirror the web's
   `lib/queries/*` and `lib/actions/*`).
2. Replace `useAsyncData` internals with the real fetch layer (it is
   deliberately React-Query-shaped — data/isLoading/error/refresh).
3. Auth: swap `AuthContext.signIn` to the credentials endpoint; property
   scoping already flows through every query as `propertyId`.
4. Files (KYC photos, receipts, invoice PDFs) go through the web's
   authenticated `/api/files/[...key]` route; `src/services` defines the
   upload interfaces.
